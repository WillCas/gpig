GPIG Interface

This is the primary defintion for the GPIG interface that all teams should work from. This interface is designed from the perspective that the virus model is the server, and the WHO model is the client making requests. Extensions and modifications are allowed at the discretion of the clients.

There are a number of general minimum requirements for the server that are assumed:

  • Requests can support a body of up to 10MB
  • Responses can support a body of up to 100MB

Each 'full turn' is a complete virus -> WHO back-and-forth. This reflects a 'ply'.

post /actions
Adjust actions in batch (postActions)

Allows the creation and deletion of actions in batch. This request will only fail when the entire request does, individual actions will each have their own response objects with dedicated status codes.

This means that if an action fails with the status code 400, then the HTTP request will respond with 200, however the action reponse object with the same index will contain a code of 400.

Consumes

This API call consumes the following media types via the Content-Type request header:
  • application/json

Request body

body (optional)
Body Parameter — The action(s) you want to create or update.

Return type

array[ActionResult]

Example data

Content-Type: application/json
[ {
  "code" : 0,
  "id" : 0,
  "message" : "message"
}, {
  "code" : 0,
  "id" : 0,
  "message" : "message"
} ]

ActionResult - Action Result Up

The result object of an action.

If an action was sent with a coordinate that does not exist, then this response will have the code 404, and the action will not be applied.

If an action failed to apply (i.e. that action has not been applied to the server), then the reponse will have the code 500. If an action has been partially applied, then these changes must be undone by the server before the response is sent.

id
Integer The ID of the action that was adjusted. This will match the ID that was sent. format: int32
code
Integer The response code for the action adjustment. This uses standard HTTP response codes. (e.g. 200 for success, 404 for an invalid location, etc.) format: int32
message (optional)
String A message containing information about the request reponse. Typically used for debug information in the case of non 2xx status codes.

LocationCoords - Location Coordinates Up

The coordinate array targeting a given location.

Coordinates are a reference to a specific grid and sub-grid in increasing in specificity. The grid is a conceptual overlay over the map to enable an abstract method of referencing sections of the map. Due to a lack of official documentation on how grid coordinates function, they are up for interpretation for specific usage in implementations.
